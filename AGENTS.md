# vue3-windows Codex Notes

## Project Structure

- `src/index.ts` is the package entrypoint.
- `src/lib/index.ts` re-exports the public API.
- `src/lib/components/Window.vue` owns dialog geometry, drag, resize, maximize, minimize, visibility, and transition behavior.
- `src/lib/components/WindowsDesktop.vue`, `WindowsDock.vue`, and `Win10Dock.vue` provide desktop and dock shells.
- `src/lib/hooks/useWindowsManager.ts` owns the window item model and imperative `WindowsRef` actions.
- `src/lib/hooks/mountWindowsRenderer.ts`, `useWindowsController.ts`, `useWindows.ts`, and `globalWindow.ts` mount windows for local desktop or standalone global use.
- `src/lib/hooks/currentWindow.ts` and `src/lib/components/WindowContentHost.ts` provide `useCurrentWindow()` inside window content.
- `test/*.spec.ts` contains Vitest coverage for window geometry, manager behavior, desktop integration, and global windows.
- `docs/` contains VitePress documentation and examples.
- `dist/` is generated by `bun run build:lib`; do not hand-edit it.

## Public API Direction

- Keep the first-use API small. Prefer `useWindows()` / `useCurrentWindow()` and `globalWindow` for standalone usage.
- When adding options, wire them through `types.ts`, `useWindowsManager.ts`, `mountWindowsRenderer.ts`, and the relevant component props.
- Preserve minimized-window dock behavior: minimized items are tracked by manager order, rendered in the dock, and restored through `moveTop`.

## Local Validation

- Single target test file: `bunx vitest run test/Window.spec.ts` with an about 10 second command timeout.
- Full test suite: `bun run test` with an about 1 minute command timeout.
- Library build: `bun run build:lib`.
- Docs build: `bun run build:docs`.
- Before release, run `bun run test`, `bun run build:lib`, and `bun run build:docs`.

## Release Flow

- Releases must run from `main`.
- Release script: `bun run release -- patch`, `bun run release -- minor`, or `bun run release -- major`.
- Dry run first when checking version/tag behavior: `bun run release -- patch --dry-run`.
- The release script updates `package.json`, regenerates `docs/changelog/index.md`, creates a `chore(release): vX.Y.Z` commit, and creates an annotated `vX.Y.Z` git tag.
- Add `--push` only when the `origin` remote is configured and pushing is intended.
- After the release commit and tag are pushed, publish the package from a clean tree with `bun publish --access public` or the npm command approved for the current environment.

## Source Editing

- Treat source and docs as UTF-8 on Windows.
- Prefer `apply_patch` for manual edits.
- Do not use PowerShell `Set-Content`, `Out-File`, or default redirection for source rewrites.
- If existing mojibake is found, do not expand it while making unrelated changes; fix encoding in a separate focused pass.
